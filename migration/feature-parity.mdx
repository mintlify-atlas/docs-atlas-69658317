---
title: "Feature parity: v1 vs v2"
description: "Comprehensive matrix of aerpawlib features across the v1 and v2 APIs, including what is new in v2 and what remains unchanged."
---

Use this page to understand what is available in each API version before choosing which one to use or deciding how much of your script to migrate.

<Info>
The v1 API is stable and receives no breaking changes. The v2 API is under active development. Both APIs ship in the same `aerpawlib` package and can coexist in the same project.
</Info>

## Vehicle types

| Feature | v1 | v2 |
|---------|----|---------|
| `Drone` | Yes | Yes |
| `Rover` | Yes | Yes |
| `DummyVehicle` | Yes | Yes |
| Blocking vehicle connection | Yes | No — connection is async (`await Drone.connect(...)`) |
| Async vehicle connection | No | Yes |

<Note>
All decorators available in v1 are also available in v2 with identical semantics. v2 implements them as descriptors (`__set_name__`) rather than inspecting function attributes at runtime, but the user-facing API is unchanged.
</Note>

---

<Warning>
Several v1 exception types — `HeadingError`, `AbortedError`, `MAVSDKNotInstalledError`, `NotInAERPAWEnvironmentError`, `ValidationError`, `InvalidAltitudeError`, `InvalidSpeedError`, `InvalidToleranceError`, `NotImplementedForVehicleError`, `StateMachineError` — are not present in the v2 public API. If your code catches these specific types, add a fallback to `AerpawlibError` when targeting v2.
</Warning>
