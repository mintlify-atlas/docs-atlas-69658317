The `examples/` directory contains working scripts that demonstrate the full range of aerpawlib patterns — from a minimal three-line mission to multi-drone ZMQ-coordinated flights. Every file runs against a real vehicle or a SITL simulator.

## All examples at a glance

| Path | Description | Demonstrates |
|------|-------------|---------------|
| `examples/v1/basic_runner.py` | Minimal takeoff → fly north → land | `BasicRunner`, `@entrypoint`, `VectorNED` |
| `examples/v1/basic_example.py` | 10 m × 10 m square flight | `BasicRunner`, `goto_coordinates`, multi-waypoint loop |
| `examples/v1/figure_eight.py` | Figure-8 via circular waypoints | Custom args, `initialize_args`, math-generated paths |
| `examples/v1/circle.py` | Circular flight using velocity control | `set_velocity`, `set_heading`, cross-product geometry |
| `examples/v1/squareoff_logging.py` | Square flight with CSV position logging | `StateMachine`, `@state`, `@background`, `@timed_state` |
| `examples/v1/preplanned_trajectory.py` | Follow a QGroundControl `.plan` file | `@at_init`, `read_from_plan_complete`, `ExternalProcess` |
| `examples/v1/hide_rover.py` | Rover follows plan, then hides inside geofence | Rover mission, geofence integration |
| `examples/v1/external_runner.py` | Spawn and read from an external process | `ExternalProcess`, `wait_until_output` |
| `examples/v1/zmq_runner/` | Leader commands a follower drone via ZMQ | `ZmqStateMachine`, `transition_runner` |
| `examples/v1/zmq_preplanned_orbit/` | Tracer + orbiter coordinated by ground station | `expose_field_zmq`, `query_field`, multi-agent patterns |
| `examples/v2/basic_example.py` | Square flight with the v2 API | v2 imports, `udpin://` connection string |
| `examples/v2/state_machine_example.py` | `StateMachine` with `@timed_state` and `@background` | v2 state machine pattern |

## v1 vs v2

<Tabs>
  <Tab title="v1 (MAVSDK-based)">
    The v1 API is the original aerpawlib interface. All production AERPAW experiments use v1.

    - Import path: `aerpawlib.v1.runner`, `aerpawlib.v1.vehicle`, `aerpawlib.v1.util`
    - Connection strings use `udp://` (e.g., `udp://127.0.0.1:14550`)
    - CLI flag: `--api-version v1` (default when omitted)

    ```python
    from aerpawlib.v1.runner import BasicRunner, entrypoint
    from aerpawlib.v1.util import VectorNED
    from aerpawlib.v1.vehicle import Drone
    ```
  </Tab>
  <Tab title="v2 (modern, async-first)">
    The v2 API is a cleaner, async-first redesign with a single top-level import.

    - Import path: `aerpawlib.v2`
    - Connection strings use `udpin://` (e.g., `udpin://127.0.0.1:14550`)
    - CLI flag: `--api-version v2`

    ```python
    from aerpawlib.v2 import BasicRunner, Drone, VectorNED, entrypoint
    ```
  </Tab>
</Tabs>

## How to run any example

<Steps>
  <Step title="Start a SITL simulator">
    Launch ArduPilot SITL or another MAVLink-compatible simulator and note the UDP port it listens on (typically `14550`).
  </Step>
  <Step title="Run the example script">
    Pass `--script` with the path to the example file, `--vehicle` with the vehicle type, and `--conn` with the MAVLink connection string.

    ```bash
    # v1 example
    aerpawlib --api-version v1 \
        --script examples/v1/basic_example.py \
        --vehicle drone \
        --conn udp://127.0.0.1:14550

    # v2 example
    aerpawlib --api-version v2 \
        --script examples/v2/basic_example.py \
        --vehicle drone \
        --conn udpin://127.0.0.1:14550
    ```
  </Step>
  <Step title="Pass extra script arguments (if required)">
    Some examples accept their own flags. Append them after the standard aerpawlib options:

    ```bash
    aerpawlib --script examples/v1/squareoff_logging.py \
        --vehicle drone --conn udp://127.0.0.1:14550 \
        --output gps_log.csv --samplerate 2
    ```
  </Step>
</Steps>

<Note>
  ZMQ multi-vehicle examples require an additional step: start the ZMQ proxy in a separate terminal with `aerpawlib --run-proxy` before launching any vehicle script. See the [ZMQ coordination](/examples/zmq-coordination) page for the full startup sequence.
</Note>
