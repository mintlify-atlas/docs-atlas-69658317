This guide walks you from a bare Python file to a drone flying a square pattern
in SITL simulation. It uses the **v1 API** — the default and most widely used
interface in aerpawlib.

## Prerequisites

- aerpawlib installed with dev/SITL extras (see [Installation](/installation))
- ArduPilot SITL configured (`aerpawlib-setup-sitl` completed)

## Write your first mission

<Steps>
  <Step title="Create a mission file">
    Create `my_mission.py` in your working directory:

    ```python my_mission.py
    from aerpawlib.v1 import Drone, Coordinate, BasicRunner, entrypoint

    class MyMission(BasicRunner):
        @entrypoint
        async def run(self, vehicle):
            await vehicle.takeoff(10)
            for wp in [
                Coordinate(35.7275, -78.6960, 10),
                Coordinate(35.7280, -78.6955, 10),
            ]:
                await vehicle.goto_coordinates(wp)
            await vehicle.land()
    ```

    This is a complete, runnable mission. The four key pieces are explained
    below.
  </Step>

  <Step title="Start ArduPilot SITL">
    In a separate terminal, start the SITL simulator:

    ```bash
    # Copter SITL on the default MAVLink UDP port
    sim_vehicle.py -v ArduCopter --out udp:127.0.0.1:14550 --console
    ```

    Wait until the console shows `APM: EKF2 IMU0 is using GPS` and the vehicle
    reports a 3D GPS fix before continuing.

    <Warning>
      aerpawlib will not arm the vehicle until it reaches an armable state. A
      missing GPS fix is the most common reason for the script to hang at
      startup.
    </Warning>
  </Step>

  <Step title="Run the mission">
    ```bash
    aerpawlib --script my_mission.py --conn udp://127.0.0.1:14550 --vehicle drone
    ```

    aerpawlib will connect, wait for the vehicle to become armable, arm it, then
    execute your `run` method. When `vehicle.land()` returns, the process exits
    cleanly.
  </Step>
</Steps>

## How the code works

### Imports

```python
from aerpawlib.v1 import Drone, Coordinate, BasicRunner, entrypoint
```

| Name | What it is |
|---|---|
| `BasicRunner` | Base class for linear, single-entrypoint missions |
| `entrypoint` | Decorator that marks the method aerpawlib calls first |
| `Drone` | Vehicle object providing all flight commands (passed in by the framework) |
| `Coordinate` | Lat/lon/alt position used by `goto_coordinates` |

### Runner class

Every mission is a class that subclasses one of the framework runner types.
aerpawlib discovers this class automatically at runtime — you do not need to
instantiate it yourself.

```python
class MyMission(BasicRunner):
    ...
```

<Note>
  There must be exactly one `Runner` subclass per script file. Defining more
  than one raises an error at startup.
</Note>

### The `@entrypoint` method

```python
@entrypoint
async def run(self, vehicle):
    ...
```

The method decorated with `@entrypoint` is the mission body. It receives the
connected vehicle as its only argument. The method is `async` — use `await` on
every vehicle command.

### Flight commands

```python
await vehicle.takeoff(10)                     # arm and climb to 10 m AGL
await vehicle.goto_coordinates(Coordinate(...))  # fly to a lat/lon/alt
await vehicle.land()                          # descend and disarm
```

Each command is awaited, so execution proceeds sequentially by default.

## Real example: square flight

The `examples/v1/basic_example.py` file included in the repository flies a
10 m × 10 m square using `VectorNED` offsets instead of absolute coordinates:

```python examples/v1/basic_example.py
from aerpawlib.v1.runner import BasicRunner, entrypoint
from aerpawlib.v1.util import VectorNED
from aerpawlib.v1.vehicle import Drone


class SquareFlight(BasicRunner):
    """Fly in a square pattern."""

    @entrypoint
    async def run(self, drone: Drone):
        takeoff_altitude = 10  # meters
        square_size = 10       # meters

        await drone.takeoff(takeoff_altitude)
        start_position = drone.position

        waypoints = [
            ("North", VectorNED(square_size, 0, 0)),
            ("East",  VectorNED(0, square_size, 0)),
            ("South", VectorNED(-square_size, 0, 0)),
            ("West",  VectorNED(0, -square_size, 0)),
        ]

        current_position = start_position
        for direction, offset in waypoints:
            target = current_position + offset
            await drone.goto_coordinates(target)
            current_position = target

        await drone.goto_coordinates(start_position)
        await drone.land()
```

Run it directly:

```bash
aerpawlib --api-version v1 --script examples/v1/basic_example.py \
    --vehicle drone --conn udp:127.0.0.1:14550
```

## Connection strings

Pass any MAVLink connection string to `--conn`:

<CodeGroup>

```bash UDP (SITL default)
aerpawlib --script my_mission.py --conn udp://127.0.0.1:14550 --vehicle drone
```

```bash UDP inbound
aerpawlib --script my_mission.py --conn udpin://127.0.0.1:14550 --vehicle drone
```

```bash Serial port (real hardware)
aerpawlib --script my_mission.py --conn /dev/ttyACM0 --vehicle drone
```

```bash TCP
aerpawlib --script my_mission.py --conn tcp://192.168.1.10:5760 --vehicle drone
```

</CodeGroup>

## Running with a config file

Instead of passing all flags on the command line, store them in a JSON file:

```json mission.json
{
  "script": "examples/v1/basic_example.py",
  "conn": "udp://127.0.0.1:14550",
  "vehicle": "drone",
  "verbose": true
}
```

```bash
aerpawlib --config mission.json
```

Any argument supplied on the command line overrides the corresponding key in
the config file.

## Running on AERPAW

When your script runs on the real AERPAW platform rather than SITL, two things
change:

<Warning>
  On AERPAW, the vehicle will **not** arm automatically. The safety pilot must
  physically arm the vehicle before aerpawlib proceeds past the pre-arm wait.
  Do not assume the mission has started until you see confirmation in the logs.
</Warning>

- Do **not** pass `--no-aerpaw-environment`. Without this flag, aerpawlib
  connects to the AERPAW platform services (OEO logging, checkpoints).
- The safety checker server is started automatically on port 14580. aerpawlib
  will refuse to proceed if it cannot reach it.

For local SITL runs, always add `--no-aerpaw-environment` to skip the platform
connection:

```bash
aerpawlib --script my_mission.py --conn udp://127.0.0.1:14550 --vehicle drone \
    --no-aerpaw-environment
```

## CLI reference (common flags)

| Flag | Description |
|---|---|
| `--script` | Path to your Python mission file |
| `--conn` | MAVLink connection string |
| `--vehicle` | `drone`, `rover`, `generic`, or `none` |
| `--api-version` | `v1` (default) or `v2` |
| `--no-aerpaw-environment` | Standalone/SITL mode — skip AERPAW platform connection |
| `--skip-init` | Skip pre-arm wait |
| `--skip-rtl` | Do not auto-RTL when the script ends |
| `-v` / `--verbose` | DEBUG logging |
| `-q` / `--quiet` | WARNING logging only |
| `--log-file` | Write logs to a file in addition to the console |
| `--conn-timeout` | Seconds to wait for initial connection (default: 30) |
| `--config` | JSON config file (keys are CLI flag names) |
